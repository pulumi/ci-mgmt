# Code Review Guidelines

Review pull requests by file type. Follow this workflow for every PR.

## Review Workflow

1. **Identify changed file types** — run `git diff --name-only` against the PR branch
2. **Skip generated files** — exclude `sdk/**` from review (see below)
3. **Review by category** — apply the rules below for each file type present
4. **Validate cross-cutting concerns** — check that patch changes align with resources.go and that CI workflows match the provider's `.ci-mgmt.yaml`

## Review Rules by File Type

### Patches (`patches/*.patch`)

See [patches.md](./patches.md) for the full review process, requirements, and commit format.

Key checks:
- New patches must link a tracking issue and have an associated test
- Modified patches may be obsolete if upstream fixed the original issue — check before approving
- Deleted patches must not break dependent patches or tests

### Provider Resources (`provider/resources.go`)

**What to check:**
- Resource and data source names follow Pulumi naming conventions (PascalCase, no `Terraform` prefix)
- Token format: `<provider>:<module>/<resourceName>:<ResourceName>`. Example: `aws:s3/bucket:Bucket` (correct) vs `aws:s3:Bucket` (wrong — missing resource name segment). Default module uses `index` as module name
- New resources have correct `Tok` assignments and are placed in the right module
- `tfbridge.ResourceInfo` fields are set appropriately (e.g., `Docs`, `Fields`, `DeleteBeforeReplace`)
- Module mappings in `tfModulesToPulumiModules` are complete — missing mappings cause `tfgen` failures
- Deprecated Terraform resources are marked with `DeprecationMessage`

**Validation:** run `make tfgen` after resources.go changes. If it fails with module mapping errors, the PR needs to add missing entries to `tfModulesToPulumiModules`.

### Generated SDK Files (`sdk/**`)

Exclude from review — fully regenerated by `make build_sdks`. Only review SDK diffs when verifying that a resources.go change produced the expected output.

### CI Workflows (`.github/workflows/**`)

Generated by ci-mgmt — manual edits will be overwritten on next sync.

**What to check:**
- Verify the change originated from a ci-mgmt update (commit message or PR reference)
- If manually edited, flag for discussion
- Check that workflow changes are consistent with `.ci-mgmt.yaml`
